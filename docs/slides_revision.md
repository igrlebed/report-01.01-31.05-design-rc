## Слайд 4 — «Дизайнер + ИИ» (граф зон ответственности)

**Было:** лента из 4 этапов ИИ-конвейера (`SlideAiNext`).

**Станет:** mind-map / граф «как в Obsidian»: в центре **Дизайнер + ИИ**, от него расходятся
задачи, сгруппированные по **отделам**, а внутри отдела видно, **функционал какого специалиста** мы закрываем.

### Главный посыл (информационный фокус)

> Не «мы используем ИИ», а **«Дизайнер + ИИ = кросс-функциональная единица»**: одна связка
> закрывает работу, которая традиционно лежит на 4 смежных отделах и функциях 5 специалистов —
> системного аналитика, UX-исследователя, frontend-разработчика, маркетолога-аналитика и SEO-специалиста.

Это и есть тот вывод, ради которого слайд существует: показать руководству **охват и рычаг** — чьи роли закрываются, а не список инструментов.

### Дерево (4 уровня)

По решению — дерево из 4 уровней. Чтобы было **стабильно** и читалось на проекторе, группируем по отделам
(отдел раньше задачи), при этом специалист — отдельный выделенный уровень, т.к. именно он и есть «вот чью работу закрываем».

```
L0  Дизайнер + ИИ                  (центр)
 └ L1  Отдел                       (4 кластера, цвет = категория)
     └ L2  Специалист              (роль, чей функционал закрываем — акцентный уровень)
         └ L3  Задача              (что именно делали, листья)
```

> Примечание: исходно ты задавал порядок «задача → отдел → специалист». Для стабильной читаемой
> картинки переставил отдел перед задачей (иначе узлы отдела/специалиста множатся или сходятся
> в кашу). Все 4 уровня сохранены, специалист остаётся ключевым акцентом. Если важен именно
> task-first порядок — пометь, поменяю.

### Анализ потока информации → раскладка дерева

Специалисты названы финально (5 ролей), задачи — с короткой подписью узла (см. копирайт ниже).

| L1 Отдел | L2 Специалист (чью работу закрываем) | L3 Задачи (короткая подпись) |
|----------|--------------------------------------|------------------------------|
| **Системный анализ** | **Системный аналитик** | • Дистилляция ТЗ<br>• Разбор формулировок<br>• Flow · IA · JTBD |
| **UX-исследования** | **UX-исследователь** | • Синтетический пользователь<br>• Проверка гипотез |
| **Разработка / Frontend** | **Frontend-разработчик** | • Кликабельный MVP<br>• UI сразу в код |
| **Маркетинг / SEO** | **Маркетолог-аналитик**<br>**SEO-специалист** | • Анализ конкурента<br>• SEO-аудит |

Итого: **1 центр → 4 отдела → 5 специалистов → 9 задач.** Достаточно «фракций» для эффектного Obsidian-вида.

**Цифра-якорь (в углу слайда):** `4 отдела · 5 специалистов · 9 задач` — усиливает посыл охвата.

### Копирайт задач (подпись узла + tooltip)

Подпись на узле — короткая (для проектора), полная формулировка раскрывается в tooltip при наведении.

| Отдел → специалист | Подпись узла | Tooltip (полная) |
|--------------------|--------------|------------------|
| Системный аналитик | **Дистилляция ТЗ** | Выжимаем из объёмного ТЗ суть — что действительно нужно сделать |
| Системный аналитик | **Разбор формулировок** | Переводим расплывчатые требования в однозначные задачи |
| Системный аналитик | **Flow · IA · JTBD** | Строим пользовательские сценарии, информационную архитектуру и Jobs-to-be-Done |
| UX-исследователь | **Синтетический пользователь** | Создаём ИИ-агента с проработанной личностью там, где реального пользователя ещё нет |
| UX-исследователь | **Проверка гипотез** | Прогоняем гипотезы через ИИ-агента и строим аналитику на его обратной связи |
| Frontend-разработчик | **Кликабельный MVP** | Проектируем интерактивные прототипы уровня MVP |
| Frontend-разработчик | **UI сразу в код** | Собираем адаптивы и приводим консистентный UI прямо в кодовую базу |
| Маркетолог-аналитик | **Анализ конкурента** | Исследуем продукт и digital-присутствие прямого конкурента → гипотезы, требования, рекомендации |
| SEO-специалист | **SEO-аудит** | Аудит сайта: расхождения с нормами и AI-ранжирование → рекомендации по выдаче в поиске и нейросетях |

### Визуальная конфигурация

**Технически:** ECharts `series.type: 'graph'`. Для **стабильности** (решение по вопросу 1) —
`layout: 'none'` с заранее заданными координатами узлов (радиальная раскладка), а не «живой» force.
Obsidian-ощущение даём за счёт стиля (круглые узлы, рёбра-пружины, свечение центра) + лёгкого
idle-«плавания» через анимацию, без риска наезжающих узлов на проекторе.

Иерархия узлов:

| Уровень | Узел | Размер | Цвет |
|---------|------|--------|------|
| L0 | «Дизайнер + ИИ» | самый крупный, свечение | брендовый `#0000BB` |
| L1 | 4 отдела | крупные | свой оттенок на отдел (`category`) |
| L2 | специалисты | средние | оттенок своего отдела, светлее |
| L3 | задачи | мелкие | приглушённый цвет отдела |

- Рёбра: центр→отдел ярче/толще, отдел→специалист средние, специалист→задача тонкие.
- Подписи: центр, отделы, специалисты — всегда; задачи — короткие (2–4 слова), полная формулировка в tooltip.

### Анимация (раскрытие графа)

1. Центр появляется и мягко пульсирует.
2. Рёбра «прорастают» к 4 отделам (stagger ~120 ms).
3. Отделы pop-in → от них к специалистам → от специалистов каскадом к задачам.
4. Постоянное лёгкое idle-«плавание» узлов (как в Obsidian), но в пределах своих позиций.

Интерактив: hover по отделу/специалисту → подсветка его поддерева, остальное приглушается.

- [x] **Force vs curated** → делаем **стабильно**: `layout:'none'` + заданные координаты (curated радиал).
- [x] **Цифра-якорь** → **да**, `4 отдела · 5 специалистов · 9 задач`.
- [x] **UX-research отдельно** → **да**, держим раздельно (4 отдела).
- [x] **Формулировки листьев** → готов копирайт: короткая подпись узла + tooltip (см. таблицу выше).
- [x] **HC-режим** → **да**, подобрать контрастные цвета отделов.
- [x] **Новое требование** → добавлен уровень «Специалист» (L2), показываем чьи роли закрываем.
- [x] **Названия специалистов** → финализированы 5 ролей (системный аналитик, UX-исследователь, frontend-разработчик, маркетолог-аналитик, SEO-специалист).

## Слайд 5 — «Нейросети: что уже сделали»

**Было:** KPI + 4 карточки проектов + график «традиционно vs с ИИ» по отдельным кейсам.

**Решение:** **каркас оставляем без изменений.** Единственная правка — логика графика: показываем
**сумму по всем прототипам сразу**, а детализацию (сколько стоил один прототип) раскрываем **на hover**.
Слайд работает как обзор и **трамплин**: на презентации звучит реплика *«рассмотрим на реальной
практике — проект ЦВСМ»* → переход на слайд 6.

### Роль слайда в сценарии

- Слайд 5 = **итог одним взглядом** (агрегат), без углубления в процесс.
- Слайд 6 = **разбор того же на примере ЦВСМ** (детальный процесс и поэтапные цифры).
- Поэтому детальное поэтапное сравнение процесса **переехало на слайд 6** (см. ниже), здесь его нет.

### График — новая логика

- **По умолчанию:** агрегированные столбцы «Традиционно vs с ИИ» — *суммарно по всем прототипам*
  (общий объём часов / экономия), а не за один прототип.
- **На hover по столбцу/кейсу:** тултип раскрывает, **сколько стоил один прототип** (часы традиционно / с ИИ, ускорение).
- Переключатель «часы / разы» — оставляем как есть (уже реализован).

> Датасет берём из `mocks/slides.json → ai.timeSaved`, но агрегируем: суммарный столбец + per-unit в тултипе.
> Точные суммы уточнить перед вёрсткой.

### Структура слайда (без изменений каркаса)

|------|--------|

## Слайд 6 — «Кейс ЦВСМ» (новый)

**Было:** одна карточка внутри слайда 5 + строка на графике.

**Станет:** **отдельный слайд-история** — самый сложный по внедрению проект как доказательство подхода «Дизайнер + ИИ».

### Лид (продающая формулировка)

> **ЦВСМ: от меняющегося расплывчатого ТЗ — до интерактивного MVP за 3–4 дня.**
> Силами одного дизайнера с ИИ — без отдела аналитики и без фронтенд-команды, которых
> традиционный процесс потребовал бы обязательно.

### Контекст (почему это сложный кейс)

ЦВСМ — проект с самым тяжёлым стартом: **ТЗ постоянно менялось** и было **расплывчатым по
функционалу** (детально описаны только стек, тех-реализация и безопасность; функционал — общими
словами, каждое требование можно трактовать по-разному). Заказчику при этом был нужен **интерактивный**
прототип, а не статичные вайрфреймы.

### Процесс: где ИИ снял нагрузку (боль → решение → эффект)

| Этап | Что было больно | Что сделали с ИИ | Эффект по времени |
|------|-----------------|------------------|-------------------|
| **1. Сравнение версий ТЗ** | ТЗ часто менялось — нужно быстро ловить, что именно изменилось | Нейросеть парсит и сравнивает версии, дизайнер анализирует только дельту | **2–3 ч → 30 мин** на версию |
| **2. Дистилляция требований** | Тех-детали забивают функционал; одно требование = разные трактовки | Парсингом убираем тех-«мусор», оставляем чистые требования к функционалу; стартовый промт дизайнера (суть продукта, область применения) держит ИИ в рамках ТЗ | Однозначные требования вместо разночтений |
| **3. Выбор направления** | Вариантов трактовки много | ИИ генерирует интерпретации → дизайнер вручную отбирает наиболее вероятные | Решение принимает человек, рутину готовит ИИ |
| **4. Проектные артефакты** | IA, user flow, JTBD — обычно работа аналитиков | ИИ строит информационную архитектуру, пользовательский путь, JTBD | Без привлечения отдела аналитики |
| **5. Дизайн-документация** | Артефакты разрозненны | Повторная дистилляция в чёткую **поэкранную** документацию со связями — база для промтов и прототипа | Готовый фундамент за ~2 дня |
| **6. Сборка прототипа** | Интерактив традиционно требует фронтендеров + аналитиков | Дизайнер + ИИ собирают интерактивный прототип | **от 2 недель и 3 отделов → 3–4 дня, 1 человек** |
| **7. Правки по фидбеку** | После показа заказчику — корректировки документации и прототипа | Правим прототип и документацию связкой дизайнер + ИИ | **от 2 дней → 2–5 часов** |

### Итог для руководства (якорные цифры)

- **Полный цикл «ТЗ → интерактивный MVP»:** традиционно от **2 недель** и **3 отдела**
  (дизайн + аналитика + фронтенд) → **3–4 дня** силами **одного** дизайнера с ИИ.
- **Правки в готовый прототип:** от **2 дней** → **2–5 часов**.
- **Сравнение версий ТЗ:** **2–3 ч → 30 мин**.
- **Не пришлось привлекать** отдел аналитики и фронтенд-команду.

### Поэтапные цифры (для сравнения traditional vs ИИ — живёт здесь)

7 шагов сворачиваем в **3 фазы** (иначе слишком дробно для одного слайда):

| Фаза | Шаги | Традиционно | С ИИ | Кого традиционно привлекали |
|------|------|------------:|-----:|------------------------------|
| **A. Разбор ТЗ** | сравнение версий + дистилляция требований | ~60 ч (от 1,5–2 нед) | ~16 ч (2 дня) | Системный аналитик |
| **B. Проектирование** | выбор направления + IA/flow/JTBD + дизайн-док | входит в фазу A традиционно | входит в 2 дня фундамента | UX-исследователь / аналитик |
| **C. Прототип + правки** | сборка интерактива + итерации по фидбеку | от 1–2 нед (+ фронтенд) | 1,5–2 дня + правки 2–5 ч | Frontend-разработчик |

Микро-вин выносим отдельной плашкой: **сравнение версии ТЗ 2–3 ч → 30 мин**.

> Цифры «традиционно» — по нижней границе («от …»), чтобы не завышать эффект. Финал уточнить перед вёрсткой.

### Инструменты

Perplexity (Sonnet 4.6, GPT 5.4) — анализ и дистилляция ТЗ; Figma Make → Cursor — сборка интерактивного прототипа.

### Связь со слайдом 4 (явная отсылка)

Слайд 6 — это **карта слайда 4 «в деле»**: фазы кейса подписываем теми же ролями, чью работу закрыла связка
(системный аналитик → UX-исследователь → frontend-разработчик). Визуальный приём: на «традиционной» дорожке
показываем чипы-роли (3 отдела), на дорожке «Дизайнер + ИИ» — один актор. Можно добавить короткую подпись-мостик:
*«те самые роли с карты — здесь на одном проекте»*.

### Предложение по формату (рекомендация)

**Вариант A — «Дорожки сравнения» (рекомендую).**
Две горизонтальные дорожки-таймлайна одна над другой, общий масштаб времени:

```
Традиционно   ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  ~2+ недели   [Сис.аналитик][UX/аналитик][Frontend]
Дизайнер + ИИ  ▓▓▓▓                    3–4 дня      [Дизайнер + ИИ]
```

- Длинная серая дорожка vs короткая брендовая — **визуальный «вау» от сжатия сроков** считывается мгновенно.
- Каждая дорожка сегментирована по 3 фазам (A/B/C) с подписями времени.
- Под фазами — чипы ролей (отсылка к слайду 4): на «традиционно» 3 отдела, на «ИИ» один актор.
- Сверху — лид; снизу — якорная строка «от 2 недель и 3 отделов → 3–4 дня, 1 человек» + плашка микро-вина (ТЗ 2–3 ч → 30 мин).
- Детали каждого шага (боль→решение) — короткие подписи на сегментах или в tooltip, чтобы не перегружать.

**Вариант B — «Вертикальный процесс + парные бары».**
Слева вертикальный список 3 фаз (с краткой сутью «что делали с ИИ»), справа — для каждой фазы пара
горизонтальных столбцов traditional/ИИ. Плотнее по тексту, но менее «киношно», чем дорожки.

**Вариант C — «Поток слева-направо».**
Этапы как карточки-ноды в ряд со стрелками (как pipeline на старом слайде 4), у каждой — мини-сравнение времени.
