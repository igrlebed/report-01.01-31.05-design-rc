# Презентация: отчёт дизайн-отдела за первое полугодие

**Цель:** за 5–7 минут на митапе показать руководству здоровье портфеля проектов и опыт внедрения ИИ.  
**Главный посыл:** команда держит сроки (85,5% дедлайнов в срок), а «провалы» в выгрузке объясняются процессом учёта, а не исполнением.  
**Период данных:** 01.01.2026 — 31.05.2026.  
**Аудитория:** топ-менеджмент → крупные цифры, минимум текста, акцент на выводах.

---

## Общая концепция

- **8 экранов:** обложка + 7 контентных слайдов.
- Каждый график — **интерактивный**: hover-подсказки, drill-down, переключатели единиц.
- Цветовой код: зелёный — в срок, красный — просрочка, брендовый `#0000BB` — акцент и «с ИИ», серый — нейтраль/традиционный подход.
- Данные — статичный JSON (`/mocks/slides.json`), агрегаты из `docs/report.md`; источники — `tracker.json` + `labor_tracking.xlsx`.
- Сборка: **Nuxt 3 + PrimeVue 4 + ECharts** (`components/slides/*`, навигация в `pages/index.vue`).

## Слайд 2 — «Дедлайны — главный показатель»

**Вопрос руководителя:** почему верить именно этой цифре?

**Шапка метрики:** `85,5%` (171/200) — агрегат после корректировок (`deadlines.totalAfter`).

**График — Horizontal bar «% в срок по проектам».**
- Проекты отсортированы по возрастанию %.
- Цвет столбца: ≥90% зелёный, ≥70% брендовый, иначе красный.
- ЕИМ 87%, ЦВСМ 98%, АС МИП 54%, продуктовые ~100%.
- Интерактив: hover → числа; **клик по проекту** → тот же drill-down с просрочками, что на слайде 1.

**Три карточки-контекста (справа):**
1. **Откуда просрочки?** — нумерованный список причин:
   - перенос сроков по низкоприоритетным задачам бэклога;
   - долгое ревью — ожидание обратной связи от аналитики;
   - корректировки требований от заказчика;
   - ожидание ответов на вопросы по ТЗ.
2. **АС МИП (54%)** — «хотелки заказчика», переделки вне контроля команды → метрику оцениваем отдельно.
3. **Оценки vs факт не используем** — на закрытии бэклога ЕИМ объём перекрывали внутри одной задачи (напр. EIMNEW-1727).

**Drill-down (слайды 1–2):** модальное окно 760px, колонка «Задача» расширена, закрытие по клику на затемнённую область.

> Переключатель «до / после корректировок» в UI не выведен; в JSON есть `deadlines.before` (78,5%) и `deadlines.after` (85,5%).

## Слайд 7 — «Нейросети: что планируем делать»

**Компонент:** `SlideAiPlan.vue`  
**Вопрос руководителя:** куда масштабируем опыт и **как поймём, что это работает**, а не просто «генерируем быстрее»?

**Ключевой посыл:** масштабирование ИИ в дизайне — не только инструменты и процесс, но и **система измерений**. ИИ смещает нагрузку с написания на проверку; без парных метрик «скорость + качество» легко получить иллюзию ускорения.

**Источник для расширения:** `docs/implementation.pdf` — трёхуровневая модель метрик, рекомендации по внедрению, KPI по отделам (дизайн / фронт / бэк), метрики по этапам SDLC.

> **Статус:** ниже — предложение по переработке слайда (без кода). Текущая вёрстка: лид + 3 шага таймлайна + декоративная графика.

### Что взять из `implementation.pdf` (и что не тащить на экран)

| Из документа | Зачем на слайде | Что показать руководству |
|--------------|-----------------|--------------------------|
| **Тезис «AI смещает нагрузку с написания на проверку»** | Объясняет, почему «просто быстрее» недостаточно | 1 строка под лидом или в блоке «принцип» |
| **3 уровня «балансирующих метрик»** | Каркас оценки эффективности | 3 плашки: Скорость · Качество · Здоровье системы |
| **Парные метрики** (скорость + качество, never lead time alone) | Защита от ложного ROI | Короткая подпись под плашками или в «принципах» |
| **Outcome, не Output** (не строки кода) | Связь со слайдами 5–6 | В шаге про прототип как фундамент |
| **Baseline + AI_Assisted в Трекере** | Конкретное действие «с чего начнём» | Шаг 1 дорожной карты или отдельная плашка «Старт» |
| **Vibe Check / Conversion Rate** (прототип → прод, порог ~30% переписывания) | Мост слайд 6 → 7 | Метрика успеха для дизайна |
| **KPI дизайна** (Time-to-First-Prototype, Design-to-Dev Friction, Consistency) | Релевантно аудитории отчёта | 2–3 KPI на слайде, остальное — в speech |
| **KPI фронта / бэка, SDLC-таблица** | Контекст для смежников | **Не на слайд** — 1 фраза «согласуем единые принципы со стеком» (уже есть в шаге 1) |

**Не выносить на экран:** детальные метрики фронтенда (Boilerplate-to-Core, FTR UI, LCP/ARIA), бэкенда (CFR, SAST/DAST), полную SDLC-таблицу — перегруз для 7-го слайда и финала презентации.

#### Блок A — Лид + подзаголовок

| Элемент | Текст (черновик) | Источник PDF |
|---------|------------------|--------------|
| Лид | **Масштабирование** | без изменений |
| Подзаголовок | ИИ смещает нагрузку с написания на **проверку** — масштабируем только то, что измеряем парными метриками | стр. 1, «трёхуровневая модель» |

#### Блок B — Дорожная карта (3 шага, переформулировать под PDF)

Текущие шаги из `ai.roadmap` сохраняем по смыслу, **уточняем формулировки** под документ:

| Шаг | Было (JSON) | Станет (предложение) | Связь с PDF |
|-----|-------------|----------------------|-------------|
| **1** | Согласовать единые фреймворки… | **Baseline и учёт:** зафиксировать Cycle Time «до ИИ» по архивным задачам; поле `AI_Assisted` в Трекере | п. 1–2 рекомендаций |
| **2** | Масштабировать на всю команду… | **На всю команду дизайнеров:** единые промты, шаблоны, ревью прототипов; не одни и те же метрики на все этапы SDLC | п. 4 рекомендаций + SDLC |
| **3** | Прототип = фундамент… | **Прототип → продакшн:** outcome, не output; целевой Conversion Rate прототипа в продукт (переписывание ≤ ~30%) | Vibe Check + `ai.evolution` |

#### Блок C — «Балансирующие метрики» (3 плашки)

Компактно, без таблиц — для топ-менеджмента:

| Плашка | Суть | Риск (микроподпись, опционально) |
|--------|------|----------------------------------|
| **Скорость** | Lead / Cycle time, time-to-first-prototype | Ускорили черновик — застряли на ревью |
| **Качество** | Rework, acceptance rate, вопросы от разработки | «Галлюцинации» и скрытый долг |
| **Здоровье** | Consistency с DS, стабильность после handoff | Раздувание кодовой базы / расхождение с системой |

**Якорная формула слайда (1 строка):** эффективное внедрение = **Lead Time ↓** при **стабильном или лучшем** качестве и rework. Если скорость растёт, а баги и правки — тоже, выигрыша нет.

#### Блок D — KPI дизайна (из PDF, только релевантное отчёту)

На слайд — **максимум 3**, остальное в `speech.md`:

1. **Time-to-First-Prototype** — от брифа до первой кликабельной версии (прямая связь со слайдами 5–6).
2. **Design-to-Dev Friction** — число уточняющих вопросов от разработки по макетам (↓ = лучше спеки / handoff).
3. **Prototype → Production** — доля прототипа, дошедшая до прод без полного переписывания; порог неэффективности **> 30%** переписывания кода.

Опционально четвёртым в speech, не на экран: **Consistency Score** (% элементов по дизайн-системе).
