# Презентация: отчёт дизайн-отдела за первое полугодие

**Цель:** за 5–7 минут на митапе показать руководству здоровье портфеля проектов и опыт внедрения ИИ.  
**Главный посыл:** команда держит сроки (85,5% дедлайнов в срок), а «провалы» в выгрузке объясняются процессом учёта, а не исполнением.  
**Период данных:** 01.01.2026 — 31.05.2026.  
**Аудитория:** топ-менеджмент → крупные цифры, минимум текста, акцент на выводах.

---

## Общая концепция

- **8 экранов:** обложка + 7 контентных слайдов.
- Каждый график — **интерактивный**: hover-подсказки, drill-down, переключатели единиц.
- Цветовой код: зелёный — в срок, красный — просрочка, брендовый `#0000BB` — акцент и «с ИИ», серый — нейтраль/традиционный подход.
- Данные — статичный JSON (`/mocks/slides.json`), агрегаты из `docs/report.md`; источники — `tracker.json` + `labor_tracking.xlsx`.
- Сборка: **Nuxt 3 + PrimeVue 4 + ECharts** (`components/slides/*`, навигация в `pages/index.vue`).

## Слайд 2 — «Дедлайны — главный показатель»

**Вопрос руководителя:** почему верить именно этой цифре?

**Шапка метрики:** `85,5%` (171/200) — агрегат после корректировок (`deadlines.totalAfter`).

**График — Horizontal bar «% в срок по проектам».**
- Проекты отсортированы по возрастанию %.
- Цвет столбца: ≥90% зелёный, ≥70% брендовый, иначе красный.
- ЕИМ 87%, ЦВСМ 98%, АС МИП 54%, продуктовые ~100%.
- Интерактив: hover → числа; **клик по проекту** → тот же drill-down с просрочками, что на слайде 1.

**Три карточки-контекста (справа):**
1. **Откуда просрочки?** — нумерованный список причин:
   - перенос сроков по низкоприоритетным задачам бэклога;
   - долгое ревью — ожидание обратной связи от аналитики;
   - корректировки требований от заказчика;
   - ожидание ответов на вопросы по ТЗ.
2. **АС МИП (54%)** — «хотелки заказчика», переделки вне контроля команды → метрику оцениваем отдельно.
3. **Оценки vs факт не используем** — на закрытии бэклога ЕИМ объём перекрывали внутри одной задачи (напр. EIMNEW-1727).

**Drill-down (слайды 1–2):** модальное окно 760px, колонка «Задача» расширена, закрытие по клику на затемнённую область.

> Переключатель «до / после корректировок» в UI не выведен; в JSON есть `deadlines.before` (78,5%) и `deadlines.after` (85,5%).

## Слайд 4 — «Нейросети: какие роли закрываем»

**Компонент:** `SlideAiNext.vue`  
**Вопрос руководителя:** какие роли в продуктовой разработке закрывает связка «дизайнер + ИИ»?

**Ключевой посыл:** один дизайнер с ИИ заменяет работу нескольких отделов — от анализа ТЗ до сборки интерактива.

**Шапка:** якорные цифры справа — **4** отдела · **5** специалистов · **9** задач (`ai.map.anchor`).

**Layout — split:**
- **Слева (42%)** — сетка из 9 карточек задач (`nx-cards`): краткий заголовок, описание, теги отдела и роли. Последняя карточка (SEO-аудит) на всю ширину.
- **Справа** — mind-map граф (ECharts `graph`): центр «Дизайнер + ИИ», ветки по отделам → специалистам → задачам.

**Интерактив:** hover на карточку подсвечивает соответствующую ветку на графе; узлы слегка «плавают» (RAF-анимация).

**Данные:** `slides.json` → `ai.map` (4 отдела: системный анализ, UX-исследования, frontend, SEO).

> Старый ИИ-конвейер из 4 шагов (`ai.pipeline`) в UI не выведен — заменён картой ролей. JSON сохранён для справки.

## Слайд 6 — «Нейросети: пример из реальной практики»

**Компонент:** `SlideAiCase.vue`  
**Вопрос руководителя:** как это работает на самом сложном проекте?

**Ключевой посыл:** даже при расплывчатом меняющемся ТЗ один дизайнер с ИИ собрал интерактивный MVP за 3–4 дня вместо 2+ недель и трёх отделов.

**Лид (Tektur 56px):** «ЦВСМ: от меняющегося расплывчатого ТЗ — до интерактивного MVP за 4 дня».

**Контекст:** почему старт был тяжёлым — ТЗ без конкретики по функционалу, нужен интерактив, а не статичные вайрфреймы.

**Блок сравнения сроков (дорожки):**
- Заголовки 3 фаз над барами.
- **Дизайнер + ИИ** (верхняя строка) — 3 сегмента + «Итого 3–4 дня», 1 человек.
- **vs**
- **Традиционно** (нижняя строка) — те же 3 фазы, «Итого от 2 недель», дизайн + аналитика + фронтенд.
- Бары с подложкой `#ececf2` — виден полный масштаб; сегменты ИИ короче традиционных.

| Фаза | Традиционно | С ИИ | Роли (традиционно) |
|------|------------:|-----:|--------------------|
| Разбор ТЗ и проектирование | от 1,5–2 недель | 2 дня | Системный аналитик, UX-исследователь |
| Сборка прототипа | от 1 недели | 1,5 дня | Frontend-разработчик |
| Правки по фидбеку | от 2 дней | 2–5 часов | Frontend + системный аналитик |

**3 карточки фаз (снизу):** название · роли · развёрнутое описание процесса · строка «*традиционно* → **с ИИ**».

**Данные:** `slides.json` → `ai.cvsm`.

**Связь со слайдом 4:** те же роли (системный аналитик, UX, frontend) — здесь на одном проекте.

## Слайд 7 — «Нейросети: что планируем делать»

**Компонент:** `SlideAiPlan.vue`  
**Вопрос руководителя:** куда масштабировать опыт?

**Ключевой посыл:** превратить ИИ-прототипы из «картинки для согласования» в **фундамент продукта**.

**Лид:** «Масштабирование».

**Дорожная карта (3 шага, вертикальный таймлайн):**
1. Согласовать единые фреймворки и принципы со стеком фронтенда.
2. Масштабировать подходы на всю команду дизайнеров.
3. Прототипы дизайнеров = готовый фундамент, а не только картинка.

Декоративная пиксельная графика справа (`/img/slide6-pixels.svg`, `#0000BB`).

> Блок «было → станет» (`ai.evolution`) в JSON есть, в UI не выведен — смысл заложен в шагах дорожной карты.

## Навигация и режимы

- **Клавиатура:** `←` / `→`, `PageUp` / `PageDown`.
- **Футер (контентные слайды):** логотип · «Назад» · счётчик `N/7` · «Вперёд».
